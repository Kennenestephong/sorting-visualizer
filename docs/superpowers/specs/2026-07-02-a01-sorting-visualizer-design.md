# Дизайн: А-01 «Визуализатор алгоритмов сортировки»

- **Дата:** 2026-07-02
- **Вариант:** А-01 (Трек А, учебная ознакомительная практика, 09.03.04)
- **Стек:** Python 3.12 + PySide6
- **Объём этого дизайна:** только приложение (код + тесты + Docker). Отчёт проектируется отдельно.

## 1. Цель и объём

Настольное приложение, визуализирующее работу четырёх алгоритмов сортировки с пошаговой
анимацией, режимом «гонки» и подсчётом операций. Данные хранятся в файлах (JSON/CSV),
развёртывание через Docker.

**Требования задания А-01, закрываемые дизайном:**

- не менее 4 алгоритмов сортировки: пузырьковая, вставками, слиянием, быстрая (ровно эти 4);
- пошаговая анимация сравнений и перестановок;
- режим «гонки» нескольких алгоритмов на одном массиве;
- настройка размера и способа заполнения (случайный, обратный, почти отсортированный);
- вывод числа операций.

**Общие требования практики, закрываемые дизайном:** хранение в файлах (JSON/CSV, без СУБД);
`Dockerfile` для сборки и запуска; юнит-тесты на ядро + интеграционные тесты + тестовые
данные; декомпозиция на модули.

### Зафиксированные решения (из брейнсторминга)

| Вопрос | Решение |
|---|---|
| Объём | Только приложение (код + тесты + Docker); отчёт отдельно |
| Набор алгоритмов | Ровно 4 (пузырёк, вставки, слияние, быстрая) |
| Управление анимацией | Play/pause, слайдер скорости, сброс, **шаг вперёд и шаг назад** (полная перемотка) |
| Работа с файлами | Массив ↔ JSON (сохранение/загрузка) + статистика прогонов → CSV |
| Режим гонки | Всегда все 4 алгоритма, 4 фиксированные панели, один общий массив |
| Qt-биндинг | PySide6 (LGPL) |
| Архитектура шагов | Подход 1: алгоритмы-генераторы, отдающие события; материализованный лог |

### Незафиксированные параметры (значения по умолчанию)

- Диапазон размера массива: 10–200, по умолчанию 50.
- Интервал `QTimer` (скорость): 5–500 мс/шаг.
- Цветовая схема: обычный — серый, сравнение — жёлтый, запись/перестановка — красный,
  отсортирован — зелёный.

## 2. Архитектура

Три слоя, зависимости строго в одну сторону: **UI → core → events**.
Ядро (`core/`, `io/`) не зависит от Qt.

```
sorting_visualizer/
  core/                     # чистый Python, без Qt
    events.py               # dataclass-события: Compare, Swap, Overwrite, MarkSorted
    algorithms/
      base.py               # протокол алгоритма + реестр (registry)
      bubble.py insertion.py merge.py quick.py
    fill.py                 # генерация массива: random / reversed / nearly_sorted
    runner.py               # прогон генератора -> list[Event]; apply/revert к состоянию
    stats.py                # подсчёт сравнений / записей / времени
  io/                       # чистый Python
    array_store.py          # save/load массива в JSON, ArrayLoadError
    stats_export.py         # экспорт статистики в CSV
  ui/                       # PySide6
    bar_widget.py           # QWidget с paintEvent — рисует столбики
    controls.py             # play/pause, шаг +/-, скорость, размер, способ заполнения
    single_view.py          # просмотр одного алгоритма
    race_view.py            # 4 панели, синхронный прогон
    main_window.py          # переключение режимов, меню Файл (save/load/export)
  app.py                    # точка входа: QApplication
tests/
  core/  io/  ui/  data/
Dockerfile  requirements.txt  README.md  .dockerignore
```

### Поток данных

Алгоритм (генератор) → `runner.record()` материализует `list[Event]` → UI держит
`step_index` и по `QTimer` двигает его вперёд/назад, применяя/откатывая события к снимку
массива (`state`) → `bar_widget` перерисовывается с подсветкой ролей.

Анимация — это чистое «проигрывание» заранее посчитанного лога. При любой смене параметров
(размер, заполнение, алгоритм) UI заново вызывает `runner.record(...)` и сбрасывает
`step_index = 0`.

## 3. Ядро

### 3.1 События (`events.py`)

Неизменяемые dataclass'ы. Минимальный набор, покрывающий все 4 алгоритма:

| Событие | Поля | Смысл | Откат |
|---|---|---|---|
| `Compare` | `i, j` | сравнили два элемента (подсветка) | само-обратимо (ничего не меняет) |
| `Swap` | `i, j` | поменяли местами | тот же swap |
| `Overwrite` | `i, value, old` | записали значение в ячейку (слияние) | вернуть `old` |
| `MarkSorted` | `i` | элемент занял финальное место | снять метку |

`old` в `Overwrite` хранится специально, чтобы «шаг назад» был точным и дешёвым.

### 3.2 Алгоритмы (`algorithms/*.py`)

Каждый алгоритм — чистая функция-генератор `(a: list[int]) -> Iterator[Event]`.

- пузырьковая, вставками, быстрая — через `Swap` (in-place);
- слиянием — через `Overwrite` (запись из вспомогательного буфера).

Все четыре регистрируются в реестре (`base.py`); UI берёт список алгоритмов оттуда.

Пример контракта:
```python
def bubble_sort(a: list[int]) -> Iterator[Event]:
    ...
    yield Compare(j, j + 1)
    yield Swap(j, j + 1)
    ...
    yield MarkSorted(k)
```

### 3.3 Runner (`runner.py`)

- `record(algorithm, array) -> list[Event]` — прогоняет генератор, собирает события,
  замеряет время самого прогона (без анимации).
- `apply(state, event)` / `revert(state, event)` — двигают снимок массива вперёд/назад.
- `state` = массив + множество «отсортированных» индексов + текущие подсветки.

### 3.4 Stats (`stats.py`)

Пробегает `list[Event]` и считает две метрики: **сравнения** (число событий `Compare`) и
**перестановки** (число событий `Swap` плюс число событий `Overwrite`; один `Swap` считается
как одна перестановка, а не две записи). **Время** прогона замеряет `runner`. В CSV колонка
называется `writes` и содержит именно эту метрику перестановок.

### 3.5 Генерация массива (`fill.py`)

Три способа: `random`, `reversed`, `nearly_sorted`. Поддержка фиксированного seed для
воспроизводимости в тестах.

## 4. UI (PySide6)

### 4.1 `bar_widget.py`

Кастомный `QWidget` с `paintEvent`, рисует массив вертикальными столбиками
(высота ∝ значению) через `QPainter`. Цвет зависит от роли в текущем шаге (см. цветовую
схему в п.1). Получает `state`, содержит только отрисовку — без логики алгоритмов.

### 4.2 `controls.py`

Одна переиспользуемая панель: ▶/⏸, ⏮ шаг назад, ⏭ шаг вперёд, ⟲ сброс/новый массив;
слайдер скорости (интервал `QTimer`); спиннер размера (10–200); выпадающий список способа
заполнения (случайный / обратный / почти отсортированный).

### 4.3 `single_view.py`

Одиночный режим: выбор алгоритма (из реестра) + один `bar_widget` + `controls` + строка
статистики (сравнения / записи / время). Один `QTimer` двигает `step_index`.

### 4.4 `race_view.py`

Режим гонки: 4 фиксированные панели (по алгоритму), у каждой свой `bar_widget` и живой
счётчик операций. **Один общий `QTimer`** на каждом тике двигает индекс шага во всех
четырёх. У каждого алгоритма своя длина лога; закончившийся замирает и подсвечивается
зелёным. Общие — один исходный массив и одна панель `controls`.

### 4.5 `main_window.py`

Переключатель режимов (одиночный ⇄ гонка) + меню **Файл**: сохранить/загрузить массив
(JSON), экспорт статистики (CSV). Смена размера/способа заполнения генерирует новый массив
и переприменяется к обоим режимам.

## 5. Файлы (I/O)

### 5.1 Массив ↔ JSON (`io/array_store.py`)

```json
{
  "version": 1,
  "size": 50,
  "fill": "nearly_sorted",
  "data": [37, 12, 5, 88]
}
```

- `save(path, array, fill)` — сериализация текущего массива.
- `load(path) -> LoadedArray` — читает и валидирует: файл существует, корректный JSON, есть
  ключ `data`, это список целых, длина в диапазоне 10–200. Ошибки → `ArrayLoadError`.

### 5.2 Статистика → CSV (`io/stats_export.py`)

Одна строка на алгоритм (в гонке — все 4, в одиночном — текущий):

```
algorithm,size,fill,comparisons,writes,time_ms
bubble,50,random,1225,932,1.4
quick,50,random,286,141,0.3
```

Заголовок пишется всегда; кодировка UTF-8; разделитель — запятая.

## 6. Обработка ошибок

Принцип: **ядро/IO бросают исключения, UI их ловит и показывает пользователю**.
Приложение не падает из-за плохого файла или ввода.

| Ситуация | Где | Поведение |
|---|---|---|
| Битый/несуществующий JSON, неверная схема, размер вне диапазона | `array_store.load` → `ArrayLoadError` | UI ловит, `QMessageBox`, состояние не меняется |
| Нет прав на запись / диск занят | `stats_export` / `array_store.save` → `OSError` | UI ловит, `QMessageBox`, операция отменяется |
| Экспорт, когда прогонов ещё нет | UI | пункт меню disabled либо сообщение «нет данных» |
| Некорректный размер с клавиатуры | `controls` (спиннер) | ограничен min/max на уровне виджета |

Кастомные исключения (`ArrayLoadError` и т.п.) живут в `io/` — UI ловит их точечно.

## 7. Тестирование

Пирамида: основной вес на чистом ядре (быстро, без GUI), сверху немного UI-тестов.

**Юнит-тесты ядра (`tests/core/`), без Qt:**

- каждый из 4 алгоритмов: результат отсортирован; события по порядку дают тот же
  отсортированный массив; `apply→revert` возвращает исходное состояние;
- `fill.py`: корректный размер каждого способа; «почти отсортированный» действительно почти
  отсортирован; воспроизводимость при фиксированном seed;
- `stats.py`: корректный подсчёт сравнений/записей на эталонных последовательностях;
- граничные случаи: пустой массив, 1 элемент, уже отсортированный, обратный, дубликаты.

**Юнит-тесты I/O (`tests/io/`):** round-trip save→load; отклонение битого JSON и размера вне
диапазона (`ArrayLoadError`); корректность CSV (заголовок + строки).

**Интеграционные/UI-тесты (`tests/ui/`, `pytest-qt`):**

- смена алгоритма/размера/заполнения вызывает `runner.record` и сбрасывает `step_index`;
- шаг вперёд/назад двигает состояние `bar_widget` предсказуемо;
- в гонке общий `QTimer` синхронно двигает 4 панели, закончившийся алгоритм замирает;
- экспорт CSV после прогона пишет ожидаемый файл (во временную директорию).

**Тестовые данные (`tests/data/`):** готовые JSON-массивы (штатные + граничные).

## 8. Docker

PySide6 — GUI, у контейнера нет экрана. Решение на двух уровнях:

1. **Тесты в контейнере — headless** через `xvfb` + системные библиотеки Qt
   (`libgl1`, `libegl1`, `libxkbcommon0`, `libdbus-1-3`). Запуск: `xvfb-run pytest`.
2. **Запуск GUI** требует проброса дисплея хоста (документируется в `README.md`):
   - Linux: `-e DISPLAY=$DISPLAY -v /tmp/.X11-unix:/tmp/.X11-unix`;
   - Windows/macOS: X-сервер (VcXsrv / XQuartz), `-e DISPLAY=host.docker.internal:0`.

```dockerfile
FROM python:3.12-slim
RUN apt-get update && apt-get install -y --no-install-recommends \
    libgl1 libegl1 libxkbcommon0 libdbus-1-3 xvfb && \
    rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
RUN xvfb-run -a pytest -q          # тесты как часть сборки — падает, если тесты красные
CMD ["python", "-m", "sorting_visualizer.app"]
```

- `.dockerignore`: `__pycache__`, `.git`, отчёт и т.п.
- `requirements.txt`: `PySide6`, `pytest`, `pytest-qt`.

Ограничение GUI-в-контейнере объективно для любого десктоп-приложения; его стоит явно
проговорить в отчёте и на защите.
